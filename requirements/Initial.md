# Requirements: Distributed Peer Inference Task Distribution Network

## Overview

A general-purpose distributed AI inference system that offloads tasks — summarization, translation, creative writing, or any text-generation workload — from requesting clients onto available provider peers in the network. Each provider device runs Gemma 4 on-device. The server acts purely as a signaling and availability registry; no task content ever touches it. Providers signal their own busy/available state to the server at the start and end of each task, enabling the server to maintain an accurate real-time view of the pool and route only to genuinely free devices.

---

## Problem Statement

On-device inference is constrained by a single device's compute. When a client needs inference but its own device is busy or lacks the model, it needs to delegate the task to another peer. The system must efficiently match tasks to available providers, track provider capacity in real time, and ensure all data is transferred directly and securely between peers — without the server ever seeing the task content.

---

## Architecture

```
[React Native Client / Svelte Web Client]
        |                   |
        |  (1) Request peers |
        v                   v
   [Node.js Server + MariaDB]
        |
        | (2) Returns available peer list + updates last_used timestamps
        v
[Peer A] <----P2P stream----> [Requesting Peer]
[Peer B] <----P2P stream----> [Requesting Peer]
        (3) Direct P2P inference streaming — no data touches server
```

| `is_provider` | BOOLEAN | Whether the peer has the model downloaded and can serve inference |

| `job_duration_ms` | INT | Total inference wall-clock time in milliseconds (submitted at task complete) |
| `tokens_generated` | INT | Number of output tokens produced (submitted at task complete) |
| `tokens_per_second` | FLOAT | Derived: `tokens_generated / (job_duration_ms / 1000)` — computed on insert |
| `status` | ENUM('in_progress', 'completed', 'failed') | Current task state |

## Encryption

### Overview

All inference data exchanged over P2P connections is end-to-end encrypted using a **hybrid RSA + AES scheme**. The server is only involved in distributing RSA public keys — it never sees plaintext content, AES keys, or ciphertext payloads.

### Key Types and Lifecycle

| Key | Algorithm | Scope | Stored |
|-----|-----------|-------|--------|
| RSA Key Pair | RSA-OAEP, 2048-bit | Per client identity — generated once on mount, persists for the session | Private key: client memory only. Public key: server DB + returned in `/peers/available` |
| AES Session Key | AES-GCM, 256-bit | **Per request-response cycle** — generated fresh for every inference request | Never stored; lives only in the scope of the active request handler |

### Why Hybrid Encryption

RSA alone cannot efficiently encrypt large payloads (text chunks). AES-GCM is fast and handles arbitrary-length data. The hybrid approach uses RSA only to securely transmit the AES key, and AES-GCM for the actual content — combining the key-distribution strengths of asymmetric crypto with the throughput of symmetric crypto.

---

### Client RSA Key Generation (On Mount)

When a client starts (web or mobile), before registering with the server:

1. Generate an **RSA-OAEP 2048-bit key pair** using the platform's native crypto API:
   - Web: `window.crypto.subtle.generateKey`
   - React Native: `react-native-rsa-native` or `expo-crypto`
2. Export the **public key** as PEM (or SPKI base64)
3. Keep the **private key in memory** — never serialise or persist it to storage
4. Include the public key in `POST /peers/register`

The RSA private key is scoped to the client session. If the page refreshes or the app restarts, a new key pair is generated and the public key is updated in the server via re-registration.

---

### Server Role in Encryption

The server's only encryption-related responsibility is **storing and returning RSA public keys**. It does not participate in encryption or decryption in any other way.

#### `POST /peers/register` — updated body
```json
{
  "peer_id": "abc-123",
  "platform": "web",
  "is_provider": true,
  "rsa_public_key": "-----BEGIN PUBLIC KEY-----\nMIIBIjAN..."
}
```

#### `GET /peers/available?count=3` — updated response
```json
{
  "providers": [
    { "peer_id": "peer-id-1", "rsa_public_key": "-----BEGIN PUBLIC KEY-----\n..." },
    { "peer_id": "peer-id-2", "rsa_public_key": "-----BEGIN PUBLIC KEY-----\n..." },
    { "peer_id": "peer-id-3", "rsa_public_key": "-----BEGIN PUBLIC KEY-----\n..." }
  ]
}
```

---

### Encrypted P2P Message Format

Every P2P message exchanged between requester and provider uses this envelope:

```json
{
  "request_id": "uuid-v4",
  "encrypted_aes_key": "<base64 — AES key encrypted with provider's RSA public key>",
  "iv": "<base64 — 12-byte AES-GCM initialisation vector>",
  "ciphertext": "<base64 — AES-GCM encrypted payload>"
}
```

- `request_id` ties together all messages belonging to a single inference job and is the key used to look up the correct AES session key on the provider side
- `encrypted_aes_key` is sent **only in the first message** of a request; streaming continuation messages omit it and reference `request_id` to retrieve the already-decrypted AES key from the request-scoped context
- `iv` is unique per message — a fresh 12-byte random value generated for every AES-GCM encryption call

---

### Full Encryption Flow

```
REQUESTER                                    PROVIDER
─────────                                    ────────

[On mount]
Generate RSA key pair
POST /peers/register { rsa_public_key }
                                             [On mount]
                                             Generate RSA key pair
                                             POST /peers/register { rsa_public_key }

[Inference request]
GET /peers/available → [ { peer_id, rsa_public_key } ]

For each provider:
  Generate AES-256-GCM key        ← scoped to this request_id only
  Generate IV (12 bytes random)
  Encrypt payload with AES key + IV  → ciphertext
  Encrypt AES key with provider's    → encrypted_aes_key
    RSA public key (OAEP)

  Send over PeerJS DataConnection:
  {
    request_id,
    encrypted_aes_key,
    iv,
    ciphertext           ← contains: { task_id, text_chunk, prompt_template }
  }

                                             Receive message
                                             Decrypt AES key using own RSA private key
                                             Store AES key in request-scoped Map:
                                               sessionMap.set(request_id, { aesKey })
                                             Decrypt ciphertext with AES key + IV
                                             Run Gemma 4 inference

                                             [Streaming response tokens]
                                             For each token chunk:
                                               Generate fresh IV
                                               Encrypt token with AES key from
                                                 sessionMap.get(request_id)
                                               Send: { request_id, iv, ciphertext }

Receive streamed token chunks
Look up AES key by request_id
  (stored in requester's request-scoped Map)
Decrypt each chunk with AES key + IV
Render tokens progressively

                                             [On inference complete]
                                             Send: { request_id, done: true }
                                             sessionMap.delete(request_id)  ← key purged

[On done signal received]
requestMap.delete(request_id)  ← key purged
```

---

### Request-Scoped Key Management

**This is a critical correctness requirement.** A provider may be serving multiple requesters simultaneously. AES keys must never be stored as module-level or component-level variables. They must live in a `Map` keyed by `request_id`, scoped to the active connection handler.

#### Provider — session map pattern
```
sessionMap: Map<request_id, { aesKey: CryptoKey }>

On first message:
  aesKey = RSA_decrypt(encrypted_aes_key, myPrivateKey)
  sessionMap.set(request_id, { aesKey })

On each message:
  { aesKey } = sessionMap.get(request_id)
  plaintext = AES_decrypt(ciphertext, aesKey, iv)

On done / connection close:
  sessionMap.delete(request_id)
```

#### Requester — session map pattern
```
requestMap: Map<request_id, { aesKey: CryptoKey, providerId: string }>

On sending request:
  aesKey = generateAESKey()
  requestMap.set(request_id, { aesKey, providerId })

On receiving streamed response:
  { aesKey } = requestMap.get(request_id)
  token = AES_decrypt(ciphertext, aesKey, iv)

On done signal:
  requestMap.delete(request_id)
```

The Maps are local variables inside the connection event handler — not exported, not shared across connections, not attached to global state.

## Server Responsibilities

The server acts **only as a signaling, registry, and metrics layer**. It does not handle task content, inference data, or output text.

### Endpoints

#### `POST /peers/register`
Called when a client connects. Registers (or updates) a peer in the database. Includes the client's RSA public key so the server can distribute it to requesters.

**Body:**
```json
{
  "peer_id": "abc-123",
  "platform": "web",
  "is_provider": true,
  "rsa_public_key": "-----BEGIN PUBLIC KEY-----\nMIIBIjAN..."
}
```

#### `GET /peers/available?count=3`
Returns a list of available provider peers. The server selects peers by **oldest `last_used` first** (fair cycling), updates their `last_used` to now, and returns their peer IDs and RSA public keys.

Eligibility filter: `is_provider = true` AND `is_active = true` AND `is_busy = false`

**Response:**
```json
{
  "providers": [
    { "peer_id": "peer-id-1", "rsa_public_key": "-----BEGIN PUBLIC KEY-----\n..." },
    { "peer_id": "peer-id-2", "rsa_public_key": "-----BEGIN PUBLIC KEY-----\n..." },
    { "peer_id": "peer-id-3", "rsa_public_key": "-----BEGIN PUBLIC KEY-----\n..." }
  ]
}
```

---

#### `POST /tasks/start`
Called by the **provider** immediately upon receiving a task over P2P, before inference begins. Marks the provider as busy and creates the task log entry with `status = 'in_progress'`.

**Body:**
```json
{
  "task_id": "uuid-v4",
  "provider_peer_id": "abc-123",
  "requester_peer_id": "xyz-456",
  "task_type": "summarization",
  "input_char_count": 4821,
  "started_at": "2025-05-17T10:23:00.000Z"
}
```

**Server actions:**
```
INSERT INTO task_logs (task_id, provider_peer_id, requester_peer_id, task_type,
                       input_char_count, started_at, status)
VALUES (...)

UPDATE peers SET is_busy = true WHERE peer_id = provider_peer_id
```

**Response:**
```json
{ "acknowledged": true }
```

---

#### `POST /tasks/complete`
Called by the **provider** immediately after inference finishes and the final token has been streamed to the requester. Marks the provider as available again and finalises the task log with output metrics.

**Body:**
```json
{
  "task_id": "uuid-v4",
  "provider_peer_id": "abc-123",
  "completed_at": "2025-05-17T10:23:47.312Z",
  "job_duration_ms": 47312,
  "tokens_generated": 512,
  "status": "completed"
}
```

**Server actions:**
```
UPDATE task_logs
SET completed_at       = completed_at,
    job_duration_ms    = job_duration_ms,
    tokens_generated   = tokens_generated,
    tokens_per_second  = tokens_generated / (job_duration_ms / 1000.0),
    status             = status
WHERE task_id = task_id

UPDATE peers SET is_busy = false WHERE peer_id = provider_peer_id
```

**Response:**
```json
{ "acknowledged": true }
```

> **Note:** If inference fails (model error, connection dropped), the provider calls `POST /tasks/complete` with `status: "failed"` and whatever partial metrics are available. The server still marks the peer as `is_busy = false` so it re-enters the available pool.

---

#### `POST /peers/heartbeat`
Periodic keep-alive sent by clients every **30 seconds** to confirm they are still active. On receipt, the server sets `last_heartbeat = NOW()`, `is_active = true`, and resets `health_check_failures = 0`.

**Body:**
```json
{ "peer_id": "abc-123" }
```

#### `GET /peers/:peer_id/health` *(server-initiated only)*
A lightweight ping endpoint exposed by each client. The server calls this when a heartbeat is overdue. The client responds with `200 OK` if alive. This endpoint is never called by other peers.

**Response:**
```json
{ "peer_id": "abc-123", "status": "alive" }
```

#### `DELETE /peers/:peer_id`
Called on graceful disconnect. Marks the peer as `is_active = false` and `is_busy = false`. Any in-progress task log for this peer is updated to `status = 'failed'`.

---

### Heartbeat & Health Check Logic

#### Client Responsibility — Heartbeat
- On mount, the client starts a **30-second interval** calling `POST /peers/heartbeat`
- On unmount / app background / page unload, the client calls `DELETE /peers/:peer_id` (graceful disconnect)
- Heartbeat interval should be cleared on unmount to prevent ghost calls

#### Server Responsibility — Health Check Scheduler
The server runs a background job every **60 seconds** to audit peers that have missed their heartbeat window.

```
Every 60 seconds:
  SELECT peers WHERE is_active = true
    AND last_heartbeat < NOW() - 90 seconds   ← missed at least 1 heartbeat cycle

  For each stale peer:
    Attempt GET /peers/:peer_id/health  (via PeerJS signaling channel or HTTP)
      ├─ Success → reset health_check_failures = 0, update last_heartbeat = NOW()
      └─ Failure → increment health_check_failures

    If health_check_failures >= 3:
      UPDATE peers SET is_active = false, is_provider = false
      (peer is removed from the available provider pool)
```

**Retry timing:** The server retries the health check up to **3 times** with a **5-second delay** between attempts before marking the peer inactive. This tolerates brief network hiccups without immediately evicting a peer.

```
Attempt 1 → wait 5s → Attempt 2 → wait 5s → Attempt 3 → Mark inactive
```

#### Re-activation
A peer marked inactive is **not deleted** from the database. If the client reconnects (calls `POST /peers/register` or `POST /peers/heartbeat` again), `is_active` and `health_check_failures` are reset and the peer re-enters the provider pool.

---

### Provider Selection Logic
- Filter: `is_provider = true` AND `is_active = true` AND `is_busy = false`
- Sort: `last_used ASC` (least recently used provider first — ensures fair rotation)
- Limit: `count` (requested number of providers)
- After selection: `UPDATE peers SET last_used = NOW() WHERE peer_id IN (...)`
  - Note: `is_busy` is **not** set here — the server sets it only when the provider itself calls `POST /tasks/start`, confirming it has actually received and begun the task

## Client Roles

Each client (web or mobile) can operate in two modes simultaneously:

### 1. Provider Mode

**Registration lifecycle:**
- On mount, the client checks whether the Gemma 4 model is already present locally (OPFS/IndexedDB on web; device filesystem on mobile)
- **If the model is present:** immediately calls `POST /peers/register` with `is_provider: true` and starts the 30-second heartbeat interval
- **If the model is absent:** registers with `is_provider: false`. When the user initiates a model download, upon successful completion the client re-registers with `is_provider: true` and begins heartbeating as a provider
- On unmount / app close: calls `DELETE /peers/:peer_id` to gracefully deregister

**Task handling lifecycle:**
1. Receives encrypted task payload over PeerJS DataConnection from a requester
2. Decrypts payload using RSA private key + AES key (see Encryption section)
3. Immediately calls `POST /tasks/start` with task metadata → server marks peer `is_busy = true`
4. Runs Gemma 4 inference locally; streams encrypted token output back to requester over the same P2P connection
5. Once the final token is sent, calls `POST /tasks/complete` with output metrics → server marks peer `is_busy = false`
6. Returns to the available pool automatically

### 2. Requester Mode
- Used when the local model is absent or the local device is already busy with its own inference
- Calls `GET /peers/available?count=N` to get idle provider peer IDs and their RSA public keys
- For each provider: generates a fresh AES session key, encrypts the task payload, opens a PeerJS DataConnection, and sends the encrypted envelope
- Receives streamed encrypted token chunks from the provider and decrypts them locally
- The requester creates the `task_id` (UUID v4) that is included in the encrypted payload; the provider uses this same `task_id` when calling `/tasks/start` and `/tasks/complete`

---

## Peer-to-Peer Data Flow (Task Request)

No task content passes through the server. All P2P payloads are end-to-end encrypted.

```
1.  Requester calls GET /peers/available?count=N
      → Server returns N idle providers: [{ peer_id, rsa_public_key }]
      → Server updates last_used timestamps

2.  Requester generates:
      - task_id (UUID v4)
      - AES-256-GCM session key (per task_id)
      - Encrypted task envelope:
          { task_id, task_type, input_text, prompt_template }
          encrypted with AES key, AES key encrypted with provider's RSA public key

3.  Requester opens PeerJS DataConnection to provider
    Sends encrypted envelope

4.  Provider receives and decrypts envelope
    Provider calls POST /tasks/start:
      { task_id, provider_peer_id, requester_peer_id,
        task_type, input_char_count, started_at }
      → Server: is_busy = true, task_logs INSERT

5.  Provider runs Gemma 4 inference
    Streams encrypted token chunks to requester via P2P
    Requester decrypts and renders tokens progressively

6.  Provider sends final { request_id, done: true } message
    Provider calls POST /tasks/complete:
      { task_id, provider_peer_id, completed_at,
        job_duration_ms, tokens_generated, status }
      → Server: is_busy = false, task_logs UPDATE
      → Server computes tokens_per_second on insert

7.  Both sides purge AES session key from request-scoped Map
```

## Supported Task Types

The system is task-agnostic at the transport layer. The task type is a metadata label only — the provider passes the full prompt to Gemma 4 as constructed by the requester. Built-in task type labels for metrics classification:

| Task Type | Description |
|-----------|-------------|
| `summarization` | Condense a longer input into a shorter form |
| `translation` | Convert input text from one language to another |
| `creative_writing` | Generate creative content from a prompt or seed |
| `other` | Any other text-generation task not covered above |

The requester constructs the full system prompt + user prompt and includes it in the encrypted payload. The provider does not interpret or modify the prompt — it passes it directly to the model.

## Task Metadata

### Submitted at task start (by provider → server)

| Field | Description |
|-------|-------------|
| `task_id` | UUID generated by the requester, passed in the encrypted payload |
| `task_type` | Label from the supported task types above |
| `input_char_count` | Character count of the raw input text (not the full prompt) |
| `started_at` | ISO 8601 timestamp captured by the provider when inference begins |

### Submitted at task complete (by provider → server)

| Field | Description |
|-------|-------------|
| `completed_at` | ISO 8601 timestamp captured when the final token is sent |
| `job_duration_ms` | Wall-clock time from inference start to final token in milliseconds |
| `tokens_generated` | Total number of output tokens produced |
| `status` | `completed` on success; `failed` if inference threw an error |

The server derives `tokens_per_second = tokens_generated / (job_duration_ms / 1000)` and stores it alongside the other metrics.

## Web Client Features (Svelte)

- **On Mount:** Generate RSA key pair → check for local model → register as provider immediately if present, otherwise register as consumer-only; start 30-second heartbeat interval
- **Provider Setup:** Download Gemma 4 model (stored in browser via OPFS or IndexedDB); on completion, re-register with `is_provider: true` and join provider pool
- **Task Submission (Requester):** Compose task (type, input text, prompt); fetch available providers; encrypt and dispatch; display streamed output progressively
- **Task Serving (Provider):** Listen for incoming P2P connections; on task receipt call `/tasks/start`; run inference; stream tokens; call `/tasks/complete`
- **Status Dashboard:** Provider availability status, current task (if busy), heartbeat indicator, task history (from local log)
- **Settings:** Task type defaults, model parameters (temperature, max tokens), number of providers to request
- **On Unmount / Tab Close:** Gracefully deregister via `DELETE /peers/:peer_id`; clear heartbeat interval; purge RSA private key from memory

## Mobile Client Features (React Native)

- **On Mount:** Generate RSA key pair → check for local Gemma 4 model → register as provider immediately if present, otherwise register as consumer-only; start heartbeat interval
- **Provider Setup:** Download quantized Gemma 4 model via LiteRT-LM; on completion, re-register with `is_provider: true`
- **Inference Engine:** Uses `react-native-litert-lm` to run Gemma 4 on-device (Android/iOS)
- **P2P:** Uses `@thyreality-inc/react-native-peerjs` for peer connections
- **Task Submission (Requester):** Same flow as web client — fetch providers, encrypt, dispatch, render streamed output
- **Task Serving (Provider):** Same lifecycle as web — receive task, call `/tasks/start`, infer, stream, call `/tasks/complete`
- **Background Serving:** Can serve inference requests when app is in background; heartbeat continues in background
- **On App Close / Background:** Gracefully deregister or pause provider status; purge RSA private key

## Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| Server never handles task content | Privacy — task input and output stay peer-to-peer only |
| `is_busy` flag set by provider on task start, cleared on task complete | Server reflects real provider state reported by the device itself, not inferred from timeouts |
| `is_busy` not set at dispatch time (`/peers/available`) | A dispatched peer may fail to connect before starting; only a confirmed `POST /tasks/start` call makes it busy |
| Provider reports failed tasks with `status: "failed"` | Ensures `is_busy` is always cleared even on error; peer re-enters pool immediately |
| Task metadata logged (no content) | Enables performance analytics (tokens/sec, job duration) without compromising user privacy |
| `tokens_per_second` computed server-side on insert | Single source of truth; avoids drift from client-side calculations |
| `last_used` cycling for provider selection | Prevents overloading a single device; distributes load fairly across the pool |
| Task-type label is metadata only; prompt is requester-constructed | Provider is a generic inference engine; task interpretation stays with the requester |
| On-device model (Gemma 4) | No cloud API costs, works offline, private |
| PeerJS for P2P | Abstracts WebRTC signaling; well-supported on both web and React Native |
| MariaDB | Lightweight relational store; sufficient for peer registry and task log metrics |
| Client-driven heartbeat (30s interval) | Lightweight keep-alive; client knows its own liveness better than the server |
| Server health check on missed heartbeat (3 retries, 5s apart) | Tolerates transient network blips; avoids premature eviction of temporarily unreachable peers |
| Re-activation without deletion | Peers that come back online are cheap to restore; preserves history |
| Register as provider on mount if model present | Maximises pool availability; no manual opt-in required |
| Register as provider immediately after model download | Zero delay between capability acquisition and pool contribution |
| Hybrid RSA + AES-GCM encryption for P2P | RSA handles key distribution; AES-GCM handles large payload throughput efficiently |
| AES key scoped per `task_id`, not global | Providers handle concurrent tasks; global keys would allow cross-session contamination |
| IV regenerated per AES-GCM message | Reusing an IV with the same key breaks AES-GCM confidentiality; fresh IV per message is mandatory |
| RSA private key in memory only, never persisted | Eliminates storage-layer attack surface; new pair generated on reconnect |
| AES key purged from Map on task completion | Limits the window of key exposure; minimises memory footprint on busy provider devices |

## Non-Functional Requirements

- **Latency:** Provider lookup and P2P connection setup should complete in < 2 seconds
- **Scalability:** Server should support hundreds of registered peers; provider selection is a single indexed query
- **Privacy:** Zero task content (input or output) stored on or routed through the server; only metadata metrics are persisted
- **Security:** All P2P payloads encrypted end-to-end with AES-256-GCM; AES keys encrypted with RSA-OAEP 2048-bit; no plaintext ever leaves the originating device
- **Accuracy of busy state:** `is_busy` is only set on confirmed `POST /tasks/start` and cleared on confirmed `POST /tasks/complete` or `failed` — preventing phantom busy states
- **Resilience:** If a provider disconnects mid-task, the requester should detect the dropped connection and re-request from another available peer; the server clears `is_busy` on graceful disconnect
- **Concurrency:** Provider-side AES session keys stored per `task_id` in a request-scoped Map — supports simultaneous tasks from multiple requesters without key collision
- **Model Size:** Gemma 4 quantized variant used to fit within mobile device constraints
